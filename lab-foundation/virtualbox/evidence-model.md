# Lab Evidence Model

A useful support lab should produce evidence, not only screenshots or configuration notes.

This model defines the minimum evidence expected from each practice scenario.

## Evidence Checklist

For each scenario, capture:

1. Symptom
   - What is the user or system experiencing?
   - Who is affected?
   - Is it isolated, repeated or widespread?

2. Scope
   - Which machine, user, service or application is affected?
   - Which systems are known to be working?
   - Which assumptions still need validation?

3. Checks Performed
   - Commands used
   - GUI locations checked
   - Logs reviewed
   - Configuration inspected
   - Connectivity or DNS tests performed

4. Findings
   - What was confirmed?
   - What was ruled out?
   - What remains uncertain?

5. Fix Or Workaround
   - What changed?
   - Why was the change safe?
   - Was the result validated?

6. Documentation Outcome
   - Ticket note
   - Escalation note
   - Knowledge base article
   - Runbook update

## Example Evidence Format

```text
Scenario:
Impact:
Affected system:
Initial symptom:
Checks performed:
Findings:
Action taken:
Validation:
Limitations:
Next documentation step:
```

## Portfolio Use

The evidence should show support thinking:

- clear symptom definition
- careful validation before assumptions
- safe troubleshooting steps
- readable handover notes
- practical documentation improvement

The goal is to demonstrate operational support maturity, not to exaggerate the lab as a production engineering platform.
